# ADR-001: 插件热部署技术决策

**Status:** ✅ Accepted
**Date:** 2026-01-01
**Decision Maker:** Architecture Team
**Related Documents:** PRD v1.0, Architecture Document v1.0

---

## Context（背景）

### 问题陈述

在OneBase v3插件系统设计过程中，**PRD文档**与**架构文档**之间存在关键矛盾：

| 文档 | 位置 | 陈述 |
|------|------|------|
| **PRD** | 第44-45行 | "🔥 **热部署能力（核心差异化）** - 插件加载/卸载无需重启OneBase服务器，生产环境零停机时间" |
| **Architecture** | 第54行、168行 | "**MVP阶段技术债**：无热部署 - 插件安装/卸载需重启服务器" |

### 为什么需要决策

1. **产品定位冲突**: PRD将热部署定位为"核心差异化能力"，是产品竞争优势
2. **技术复杂度**: 热部署显著增加系统复杂度（ClassLoader生命周期、Spring容器动态管理、内存泄漏风险）
3. **市场预期**: 企业客户期望生产环境零停机
4. **开发成本**: 实现热部署可能增加30%-50%开发时间

### 业务背景

**目标客户:**
- 金融、政府、制造等大型企业
- **私有化部署**，非SaaS多租户模式
- 生产环境停机成本高，但**允许计划性维护窗口**

**使用场景:**
- 插件安装/升级频率: 预估**每月1-2次**（非频繁操作）
- 维护窗口: 大多数企业有**凌晨维护窗口**（如2:00-4:00 AM）
- 用户体验: 插件功能为**增量能力**，非核心业务连续性依赖

## Decision（决策）

### ✅ 最终决策

**OneBase v3.0 MVP阶段采用分阶段策略:**

#### Phase 1 (MVP): 无热部署
- **插件安装/卸载需要重启OneBase服务器**
- 通过**优雅停机 + 滚动升级**降低停机影响
- 预计停机时间: **2-5分钟**（包含启动时间）

#### Phase 2: 可选热部署
- **基于Phase 1实际使用反馈决定是否实现热部署**
- 评估维度:
  - 插件安装频率是否超过预期（如>每月4次）
  - 客户对停机时间的实际容忍度
  - 技术团队对热部署复杂度的接受度

### 决策依据

#### 支持无热部署的理由

1. **技术复杂度显著降低**
   - 避免类加载器内存泄漏问题
   - 避免Spring Bean动态生命周期的边界情况
   - 降低测试难度（无需测试频繁加载/卸载）

2. **MVP验证优先**
   - **核心价值是插件扩展能力**，而非热部署
   - 先验证插件生态是否被接受，再优化运维体验
   - 符合"先走通，再优化"的精益原则

3. **企业客户容忍度**
   - 私有化部署客户通常有维护窗口
   - 插件为增量功能，非核心业务依赖
   - **2-5分钟计划内停机**可接受（对比OS补丁、数据库升级）

4. **技术替代方案**
   - **优雅停机 + 滚动升级**: 零数据丢失，平滑过渡
   - **多实例部署**: 通过负载均衡实现零停机升级
   - **蓝绿部署**: 企业级部署最佳实践

#### 支持热部署的理由（Phase 2考虑）

1. **竞争优势**
   - PRD定位为"核心差异化"
   - 竞品可能强调热部署能力

2. **开发体验**
   - 插件开发者本地调试更便捷
   - 无需频繁重启服务器

3. **未来扩展**
   - 高频插件使用场景（如插件市场）
   - SaaS化后无法要求停机

### 技术方案对比

| 方案 | 优势 | 劣势 | MVP适用性 |
|------|------|------|-----------|
| **无热部署（推荐）** | ✅ 实现简单<br>✅ 无内存泄漏风险<br>✅ 测试覆盖容易 | ❌ 需要停机2-5分钟 | ⭐⭐⭐⭐⭐ |
| Spring Hot Swap | ✅ 无停机<br>✅ Spring官方支持 | ❌ 仅支持方法体修改<br>❌ 不支持新增Class/Field<br>❌ 限制太多 | ⭐⭐ |
| JRebel | ✅ 强大热部署能力 | ❌ 商业许可费用高<br>❌ 增加技术栈依赖 | ⭐⭐ |
| 自定义ClassLoader | ✅ 完全控制 | ❌ 实现复杂度极高<br>❌ 内存泄漏风险大<br>❌ 开发成本+50% | ⭐⭐⭐ |
| OSGi (Apache Felix) | ✅ 成熟标准<br>✅ 动态加载/卸载 | ❌ 学习曲线陡峭<br>❌ Spring集成复杂<br>❌ 调试困难 | ⭐⭐⭐ |

## Consequences（后果）

### ✅ 正面影响

1. **降低技术风险**
   - 避免ClassLoader内存泄漏（常见且难排查）
   - 简化Spring容器生命周期管理
   - 单元测试覆盖率更容易达到80%

2. **加快MVP交付**
   - 预计节省**2-3周开发时间**
   - 减少复杂度，代码质量更高
   - 更专注于核心插件能力打磨

3. **运维实践标准化**
   - 优雅停机成为标准流程（有益于系统稳定性）
   - 强制实施蓝绿部署/滚动升级（企业级最佳实践）
   - 插件变更需要审批流程（降低风险）

### ⚠️ 负面影响

1. **产品定位调整**
   - PRD需要移除"热部署"作为核心差异化
   - 需要重新定义竞争优势（如：完整扩展点、企业级隔离、开发者友好）
   - 竞品对比时需要说明设计哲学（稳定性 > 灵活性）

2. **客户体验下降**
   - 插件安装需要停机2-5分钟
   - 需要协调维护窗口
   - 紧急Bug修复需要重启（非即时生效）

3. **开发体验影响**
   - 插件开发者本地调试需要重启应用
   - 但可通过**DevTools自动重启**缓解（Spring Boot DevTools重启时间<5秒）

### 🔧 缓解措施

1. **优化停机时间**
   - 应用启动优化: 目标**<60秒**（当前预估90-120秒）
   - 优雅停机: 等待请求完成（最长30秒超时）
   - 数据库迁移自动化

2. **部署自动化**
   - CI/CD集成: 一键部署插件
   - 健康检查: 自动验证启动成功
   - 回滚机制: 部署失败自动回退

3. **开发体验优化**
   - Spring Boot DevTools集成: 本地开发自动重启
   - 插件Mock测试框架: 无需启动完整应用
   - 快速启动模式: 跳过非必要模块

4. **文档透明化**
   - PRD明确说明MVP阶段无热部署
   - 提供Phase 2热部署路线图
   - 运维文档详细说明停机流程

## Implementation Plan（实施计划）

### Phase 1 (MVP) - 无热部署

**Week 1-2: 核心框架开发**
- [ ] PluginClassLoader实现（不支持卸载）
- [ ] PluginApplicationContext初始化（应用启动时）
- [ ] 插件生命周期: 安装 → 启动 → 停止

**Week 3: 部署流程**
- [ ] 插件上传到`plugins/`目录
- [ ] 重启应用后自动扫描和加载
- [ ] 健康检查接口验证插件状态

**Week 4: 优化**
- [ ] 应用启动时间优化（目标<60秒）
- [ ] 优雅停机实现
- [ ] 部署文档编写

### Phase 2 (可选) - 热部署评估

**评估触发条件（满足任一即启动）:**
1. MVP上线后，插件安装频率 > 每月4次
2. 3个以上客户明确要求热部署
3. 竞品压力导致丢单

**技术预研（2周POC）:**
1. Spring动态BeanDefinition注册可行性
2. ClassLoader卸载内存泄漏测试
3. 性能基准测试（加载/卸载时间）

**决策点:**
- ✅ 继续 → 实现热部署（预计4-6周）
- ❌ 放弃 → 保持无热部署，优化部署流程
