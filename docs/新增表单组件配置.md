# 给组件增加一个配置（自注册模式）

本文记录在 `app-builder` 中为配置面板新增或接入组件配置的标准流程。适用于 `MaterialConfiger/Attributes/components/*/index.tsx` 下的所有配置组件。

## 前提条件
- 组件文件采用自注册模式：每个组件在自身文件末尾调用注册函数。
- 入口使用自动加载：`apps/app-builder/src/pages/Editor/components/config/components/MaterialConfiger/Attributes/registry.tsx` 通过 `import.meta.glob('./components/**/index.tsx', { eager: true })` 载入所有组件模块。

## 新增或接入配置的步骤
1. 确认配置类型
   - 在 `@onebase/ui-kit` 中查找并选用对应的 `CONFIG_TYPES` 常量，例如：`CONFIG_TYPES.AUTO_CODE_RULES`、`CONFIG_TYPES.VERIFY`、`CONFIG_TYPES.CAROUSEL` 等。

2. 在组件文件中引入注册 API 与类型常量
   - 位置示例：`apps/app-builder/src/pages/Editor/components/config/components/MaterialConfiger/Attributes/components/YourConfig/index.tsx`
   - 引入：
     ```ts
     import { CONFIG_TYPES } from '@onebase/ui-kit';
     import { registerConfigRenderer } from '../../registry';
     ```
3. 在文件末尾添加自注册调用
   - 透传上下文参数（`id`, `handlePropsChange`, `item`, `configs`，如组件需要还包括 `handleConfigsChange` 等）：
     ```tsx
     export default YourConfig;

     registerConfigRenderer(
       CONFIG_TYPES.YOUR_TYPE,
       ({ id, handlePropsChange, item, configs, handleConfigsChange }) => (
         <YourConfig
           id={id}
           handlePropsChange={handlePropsChange}
           item={item}
           configs={configs}
           handleConfigsChange={handleConfigsChange}
         />
       )
     );
     ```
   - 一个文件可根据需求注册多个类型：重复调用 `registerConfigRenderer(CONFIG_TYPES.XXX, ...)`。
4. 保持与现有组件风格一致
   - 参考同目录下已实现的组件（如 `DynamicButtonRadioConfig`, `DynamicAutoCodeConfig`）。
   - 避免在组件内引入与工程不一致的库或风格。

## 渲染入口与加载机制
- 入口文件：`apps/app-builder/src/pages/Editor/components/config/components/MaterialConfiger/Attributes/registry.tsx`
  - `registerConfigRenderer(type, renderer)`：将渲染器写入注册表。
  - `renderConfigItem(ctx)`：根据 `ctx.item.type` 获取并渲染对应组件。
  - `const modules = import.meta.glob('./components/**/index.tsx', { eager: true });`：确保每个组件模块被加载，从而执行其文件末尾的注册。

## 示例
- 以自动编号规则为例（`DynamicAutoCodeConfig`）：
  - 组件路径：`apps/app-builder/src/pages/Editor/components/config/components/MaterialConfiger/Attributes/components/DynamicAutoCodeConfig/index.tsx`
  - 末尾注册示例：
    ```tsx
    import { getPopupContainer, useAppEntityStore, CONFIG_TYPES } from '@onebase/ui-kit';
    import { registerConfigRenderer } from '../../registry';

    // ... 组件实现 ...

    export default DynamicAutoCodeConfig;

    registerConfigRenderer(
      CONFIG_TYPES.AUTO_CODE_RULES,
      ({ id, handlePropsChange, handleConfigsChange, item, configs }) => (
        <DynamicAutoCodeConfig
          id={id}
          handlePropsChange={handlePropsChange}
          handleConfigsChange={handleConfigsChange}
          item={item}
          configs={configs}
        />
      )
    );
    ```

## 验证与工具
- 本地验证：运行 `app-builder`，在编辑器属性面板中选择对应类型，确认能渲染并交互。
- 代码检查：
  - `pnpm -C apps/app-builder lint`
  - `pnpm -C apps/app-builder build`

## 常见问题
- 报错 “Cannot access 'registerConfigRenderer' before initialization”
  - 由循环依赖初始化顺序引起。当前已在 `registry.tsx` 使用函数声明与全局懒初始化注册表解决，一般无需额外处理。
- 新类型未渲染
  - 核对 `item.type` 与注册使用的 `CONFIG_TYPES.XXX` 是否一致。
  - 确认注册调用位于组件文件末尾且模块能被入口自动加载。

## 在组件 schema.ts 中增加配置与实现逻辑

- 目标位置：在 `@onebase/ui-kit` 组件的 `schema.ts` 中定义可配置项与默认值，例如：`packages/ui-kit/src/components/Materials/Basic/FormComponents/YourComponent/schema.ts`。
- 步骤概览：
  - 定义 `Schema`、`EditData`、`Config` 三部分结构。
  - 在 `editData` 中新增一条配置项，指定 `key`、`name`、`type`，必要时提供 `range` 供单选/多选使用。
  - 在 `config` 中补充该 `key` 的默认值，保障未配置时的运行行为。

### 模板结构

```ts
import {
  baseConfig,
  baseDefault,
  layoutConfig,
  statusConfig,
  widthConfig,
  type ICommonBaseType
} from '../../../common';
import { CONFIG_TYPES, STATUS_OPTIONS, STATUS_VALUES, WIDTH_OPTIONS, WIDTH_VALUES, LAYOUT_OPTIONS, LAYOUT_VALUES } from '../../../constants';
import type { ILabelConfigType, ILayoutConfigType, IStatusConfigType, IWidthConfigType, TBooleanDefaultType, TTextDefaultType, TSelectDefaultType } from '../../../types';

export interface XYourComponentSchema {
  editData: TXYourComponentEditData;
  config: XYourComponentConfig;
}

export type TXYourComponentEditData = Array<
  | ILabelConfigType
  | IStatusConfigType<any>
  | ILayoutConfigType<any>
  | IWidthConfigType<any>
  | ICommonConfigType
>;

export interface XYourComponentConfig extends ICommonBaseType {
  label: {
    text: TTextDefaultType;
    display: TBooleanDefaultType;
  };
  status: TSelectDefaultType<any>;
  width: TSelectDefaultType<any>;
  layout: any;
  yourConfigKey?: ICommonConfigType
}

const XYourComponent: XYourComponentSchema = {
  editData: [
    ...baseConfig,
    { key: 'label', name: '标题', type: CONFIG_TYPES.LABEL_INPUT },
    {
      key: 'yourConfigKey',
      name: '你的配置名称',
      type: CONFIG_TYPES.YOUR_TYPE,
      range: [
        { key: 'optA', text: '选项A', value: 'A', default: true },
        { key: 'optB', text: '选项B', value: 'B' }
      ]
    },
    statusConfig,
    layoutConfig,
    widthConfig
  ],
  config: {
    ...baseDefault,
    label: { text: '你的组件', display: true },
    status: STATUS_VALUES[STATUS_OPTIONS.DEFAULT],
    width: WIDTH_VALUES[WIDTH_OPTIONS.HALF],
    layout: LAYOUT_VALUES[LAYOUT_OPTIONS.VERTICAL],
    yourConfigKey: 'A'
  }
};

export default XYourComponent;
```

### 说明与约定

- `editData` 描述属性面板展示项，`type` 必须是统一的 `CONFIG_TYPES` 常量；`range` 支持通过 `default: true` 标记默认选项。
- `config` 提供运行与渲染的默认值，属性面板修改会写回该对象对应键。
- 属性面板侧实现参考自注册模式，读取与写入统一使用 `handlePropsChange(item.key, value)`。

### 示例：数据选择方式（下拉/弹窗）

- 参考路径：`packages/ui-kit/src/components/Materials/Basic/FormComponents/DataSelect/schema.ts`。
- 在 `editData` 中新增项：

```ts
{
  key: 'selectMethod',
  name: '数据选择方式',
  type: CONFIG_TYPES.DATA_SELECT_MODE,
  range: [
    { key: 'dropdown', text: '下拉框', value: 'dropdown', default: true },
    { key: 'modal', text: '弹窗', value: 'modal' }
  ]
}
```

- 在 `config` 中设置默认值：

```ts
selectMethod: 'dropdown'
```

- 属性面板组件使用单选渲染（例如 `DynamicButtonRadioConfig`），会在未设置值时读取 `range` 标记的默认项并写回到 `config`。
